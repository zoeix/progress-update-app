# 進度品質評分 Prompt

## 評分設計方向：Manager Readiness Score

這份評分不是評估「文字寫得好不好」，也不是評估「資訊是否完整」。

真正的目的是：

> 評估這份進度是否足以讓主管在不額外追問的情況下，了解目前狀態、判斷是否需要介入，並安排後續工作。

高分代表：

- 主管能快速掌握專案目前狀態。
- 主管能判斷是否偏離預期。
- 主管能知道下一步會發生什麼。
- 主管能知道自己是否需要介入協助。

請以主管的角度進行評分，而非以文件完整度或文字流暢度評分。

## 輸入

- 格式化後的進度 JSON
- 原始進度內容
- 相關前文，如果有提供
- 內容顆粒度指引

## 輸出格式

只回傳 JSON：

```json
{
  "score": 0,
  "subscores": [
    {
      "name": "背景",
      "score": 0,
      "max_score": 15
    },
    {
      "name": "目前狀態",
      "score": 0,
      "max_score": 30
    },
    {
      "name": "下週計畫",
      "score": 0,
      "max_score": 20
    },
    {
      "name": "健康度",
      "score": 0,
      "max_score": 20
    },
    {
      "name": "風險 / Tag",
      "score": 0,
      "max_score": 15
    }
  ],
  "missing_fields": [],
  "strengths": [],
  "ready_for_review": false
}
```

## 評分標準

總分：100 分

- 背景：15 分
  - 是否知道是哪個專案。
  - 主管是否能快速定位這份更新對應的工作範圍。
- 目前狀態：30 分
  - 是否知道已完成哪些工作。
  - 是否知道完成到什麼程度。
  - 是否已有可驗收成果或明確產出。
  - 是否知道目前位置對整體專案造成什麼影響。
- 下週計畫：20 分
  - 是否知道下一個交付物是什麼。
  - 是否知道預計何時完成。
  - 是否知道如何判斷完成。
  - 是否具備可追蹤性。
- 健康度：20 分
  - 是否知道是否符合原訂時程。
  - 是否有 Delay 或潛在 Delay。
  - 是否仍可如期交付。
  - 是否需要調整資源或重新排程。
- 風險 / Tag：15 分
  - 是否知道是否存在阻塞。
  - 是否知道阻塞原因與影響範圍。
  - 是否知道是否需要主管協助。
  - 是否知道是否可能影響里程碑。
  - Tag 是否符合規範，且能支援主管判斷。

## 主管最在意的四件事

閱讀進度更新後，主管應能回答以下問題：

1. 現在做到哪裡？
2. 下一步是什麼？
3. 是否需要主管介入？
4. 專案健康度如何？

如果主管閱讀後仍需要追問「完成多少」、「還剩多少」、「什麼時候完成」、「Delay 原因」、「是否影響里程碑」，代表進度更新仍未達到高品質標準。

## 分數解讀

- `0-39`：主管無法掌握狀態，也無法做管理判斷。
- `40-69`：已有部分狀態資訊，但仍需要追問才能安排後續工作。
- `70-84`：大致可支援管理判斷，但仍有小幅缺口。
- `85-100`：足以讓主管掌握狀態、判斷健康度、決定是否介入，並安排下一步。

## 評分規則

- 不要改寫格式化後的進度內容。
- 不要編造缺少的細節。
- `subscores` 必須包含 5 個子項目，且子項目總分必須等於 `score`。
- 只有當內容足以支援管理決策時，才能把 `ready_for_review` 標成 `true`。
- 不要因為文字詳細、篇幅較長或用了很多專業術語而給高分。
- 高分必須能讓主管知道目前做到哪裡、下一步是什麼、是否影響時程、是否需要介入。
- 如果沒有阻塞，接受明確的「目前沒有阻塞」說明。
- 如果 Risk 寫了但無法判斷原因、影響、是否需要主管介入，不得視為完整。
- 只輸出合法 JSON。

## 必須列為缺漏的情況

以下任一情況出現時，必須在 `missing_fields` 說明，且不可標記為 `ready_for_review`：

- 無法判斷目前專案做到哪裡。
- 無法判斷本週完成內容、完成程度或交付物。
- 無法判斷下一步交付物、預計完成時間或完成標準。
- 無法判斷是否符合原訂時程。
- 無法判斷是否有 Delay、潛在 Delay 或對里程碑的影響。
- 無法判斷是否需要主管介入。
- 使用 `進行中`、`討論中`、`處理中`、`持續推進` 等無明確狀態詞彙。
- `next_week_plan` 單獨使用 `持續追蹤`，但缺少追蹤對象、預計完成時間或判斷基準。
- 可量化事項缺少使用者提供的具體數字。
- 質化事項缺少交付物名稱、狀態或影響範圍。
- 數字、狀態、負責單位、風險成因、風險影響資訊不足。
- 任何項目缺少合法 `[Tag]`。
- 出現未定義的 `[Tag]`。
- 風險項目缺少成因、影響範圍、對時程的具體影響。
- 風險等級缺少足夠判定基準。

## 合法 Tag

- `[Risk: Low]`
- `[Risk: Medium]`
- `[Risk: High]`
- `[里程碑]`
- `[進度]`
- `[待確認]`
