# 追問策略

## 提問設計

提問用來補出使用者知道、但目前週報沒有寫出的重要事實。只針對「完整度」、「支持度」、「計劃性」產生 `type: "question"` 的項目。

| 維度 | 目的 | 可追問資訊 |
| :--- | :--- | :--- |
| 完整度 | 補齊讀者理解進度所需的基本事實，讓主管不用追問才能知道這件事在講誰、做什麼、到哪裡。 | 對象、交付物、時間、目前狀態、下一步、待確認事項。 |
| 支持度 | 補上可支撐進度判斷的證據，讓描述從主觀敘述變成可判斷的資訊。 | 數字、完成比例、處理數量、影響範圍、事實依據、風險、限制、阻塞原因。 |
| 計劃性 | 補齊後續安排，讓主管知道接下來是否能追蹤、推進或協調。 | 下一步工作、預計完成時間、負責方、待確認事項、依賴關係、時程風險。 |

## 建議設計

建議用來改善文字表達方式。只針對「顆粒度」、「解釋性」產生 `type: "suggestion"` 的項目；如果需要補充未知事實，請改成提問，不要用建議替使用者編造資訊。

| 維度 | 目的 | 可調整 | 避免 | 範例 |
| :--- | :--- | :--- | :--- | :--- |
| 顆粒度 | 讓內容符合顆粒度子項：具體名稱、決策資訊、避免空泛、避免流水帳、層級適中，使主管能快速判斷進度。 | 建議補上文件名稱、功能名稱、模組名稱、單位名稱、交付物名稱或決策項目；建議把模糊字眼改成具體狀態；建議把過細操作整理成主管視角的進度句。 | 不要憑空指定不存在的文件、單位、功能或數字；不要只說「請寫具體一點」；不要把內容改得比原文更細碎。 | 建議將「處理 onboarding 欄位」改成「完成 Onboarding 欄位盤點表中的客戶基本資料欄位確認」。 |
| 解釋性 | 讓內容符合解釋性子項：背景、目的、影響、術語轉譯、判斷依據，使非專案成員也能理解這件事的意義。 | 建議補一句背景、目的、影響、受影響對象或判斷依據；建議把技術詞或內部詞改成流程、使用者或專案影響的說法。 | 不要誇大價值；不要替使用者編造不存在的成果；不要把每個技術詞都硬轉成商業口號。 | 建議在「完成 API timeout 調整」後補充「此調整可避免客服查詢客戶資料時畫面卡住，降低人工查詢成本」。 |

## 輸出要求

1. 最多產出5個問題
2. 產生問題時：
    - `type` 必須是 `question` 或 `suggestion`。
    - `tag` 必須標示該問題主要對應的評分維度，只能是 `完整度`、`顆粒度`、`支持度`、`解釋性` 或 `計劃性`。
    - `question` 必須是使用者可以直接回答的具體問題。
    - `example` 一個簡單的回答範例。
    - `priority` 只能是 `high`、`medium` 或 `low`。
    - `target_field` 必須是 `this_week_progress`、`next_week_plan`、`project_name` 或 `notes`。
    - 不要產生抽象、複合、需要使用者自行分析管理意涵的問題。
