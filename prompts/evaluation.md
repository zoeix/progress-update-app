# 進度品質評分 

## 評分指標
| 指標 | 配分 | 評核核心重點 |
| :--- | :--- | :--- |
| **1. 完整度 (Completeness)** | 10分 | 提供的內容是否完整包含對象、交付物、時間、目前狀態、下一步或待確認事項，讓讀者不用再追問基本背景。 |
| **2. 顆粒度 (Granularity)** | 20分 | 是否提供主管判斷進度所需的具體資訊，例如文件名稱、系統模組、功能名稱、單位名稱、決策項目或交付物名稱；避免使用「處理中」「優化中」「討論中」這類模糊字眼，也避免寫到過細的操作步驟或流水帳。 |
| **3. 支持度 (Support)** | 30分 | 是否有數字、事實、範圍、依據、風險或限制條件支撐說法，例如完成比例、處理數量、影響範圍、阻塞原因。 |
| **4. 解釋性 (Explanatory Power)** | 20分 | 是否說明這件事的背景、原因、影響或判斷依據，讓非技術或非專案成員也能理解為什麼重要。 |
| **5. 計劃性 (Planning)** | 20分 | 是否清楚交代下一步要做什麼、預計何時完成、由誰或哪個單位負責、目前有哪些待確認或可能影響排程的因素。 |

## 評分參考指南

*   **90-100分 (卓越)：** 對象、交付物、狀態、數據、風險與下一步都清楚，主管無須追問即可判斷進度。
*   **75-89分 (良好)：** 結構清楚且有具體資訊，但支持資料、風險說明或後續計劃可再補強。
*   **60-74分 (及格)：** 基本交代完成，但仍有資訊偏模糊、缺少依據、缺少負責單位或下一步不夠明確的問題。
*   **低於60分 (待改進)：** 內容過於空泛或流水帳，缺少具體對象、交付物、依據、風險或後續安排，讀者需要大量追問。

## 輸出格式

`summary_sentences` 必須提供剛好三句話，依序總結：整體評分判斷、主要優點、最需要補強的地方。

只回傳 JSON：

```json
{
  "score": 0,
  "summary_sentences": [
    "整體評分判斷。",
    "主要優點。",
    "最需要補強的地方。"
  ],
  "subscores": [
    {
      "name": "完整度",
      "score": 0,
      "max_score": 10
    },
    {
      "name": "顆粒度",
      "score": 0,
      "max_score": 20
    },
    {

      "score": 0,
      "max_score": 30
    },
    {
      "name": "解釋性",
      "score": 0,
      "max_score": 20
    },
    {
      "name": "計劃性",
      "score": 0,
      "max_score": 20
    }
  ],
  "missing_fields": [],

  "ready_for_review": false
}
```
