# 追問策略設計（Manager Readiness）

## 核心目的

追問不是要使用者重新分析整個專案，也不是要求使用者回答抽象管理問題。

追問的目的，是讓使用者能用一句具體內容補進週報，使主管更容易判斷：

- 目前做到哪裡。
- 下一步是什麼。
- 是否影響時程。
- 是否需要主管介入。

問題必須讓使用者知道「要補什麼內容」，而不是讓使用者猜測該怎麼回答。

## 問題設計原則

### 1. 一題只問一件事

禁止在同一題中同時詢問多個判斷。

不要：

> 目前是否仍能依 7 月初專案 Kick off 時程推進？若不能，是否會影響里程碑或需要主管協助？

這題同時問了：

- 是否依時程。
- 是否影響里程碑。
- 是否需要主管協助。

使用者很難判斷要怎麼回答。

可以改成：

> 請補充目前進度是否符合原訂時程，例如「目前仍符合時程」或「已延後，預計影響 XX 交付」。

### 2. 問題要提供回答格式

每個問題都應提供簡短回答範例，讓使用者可以直接照格式補進進度內容。

例如：

> 請補充目前完成程度，例如「已完成 API 串接，剩 QA 驗收」。

> 請補充下週預計交付物，例如「下週預計交付測試版頁面」。

> 請補充是否需要主管協助，例如「目前無需主管介入」或「需要主管協調設計資源」。

### 3. 問題要指向可寫進週報的文字

追問應要求使用者補一段可以直接放進 `this_week_progress` 或 `next_week_plan` 的內容。

不要問：

> 主管是否需要介入？

可以問：

> 請補充是否需要主管協助，例如「目前無需主管介入」或「需要主管協調 ClickUp 權限」。

### 4. 避免抽象管理語言

除非使用者原文已有明確上下文，否則避免直接使用以下問法：

- 是否影響里程碑？
- 專案健康度如何？
- 是否需要主管介入？
- 是否仍能依原訂時程推進？

這些可以作為評估目標，但不適合作為直接問題。

應改成更具體、可回答的問法：

- 請補充目前是否如期。
- 請補充若有延後，延後原因與影響範圍。
- 請補充目前是否需要主管協助。
- 請補充下一個可驗收交付物。

## 問題優先順序

追問時，依序補足最能提升主管判斷能力的資訊：

1. 目前完成程度或可驗收成果。
2. 是否如期，若延後則補充原因與影響。
3. 下一個交付物與預計完成標準。
4. 是否有阻塞或需要主管協助。
5. 風險成因與影響範圍。
6. Tag 或格式問題。

## Priority 定義

### High

缺少後主管無法判斷是否需要介入或是否影響交付。

例如：

- 完成程度不明。
- 是否如期不明。
- 延後原因不明。
- 是否需要主管協助不明。
- 阻塞或風險影響不明。

### Medium

缺少後主管仍可大致理解，但無法完整追蹤。

例如：

- 缺少交付物名稱。
- 缺少完成標準。
- 缺少影響模組或範圍。

### Low

僅補充背景，不直接影響主管決策。

例如：

- 技術實作方式。
- 使用哪個 framework。
- debug 過程。

## 追問數量

- 最多追問 3 題。
- 優先問 1 題最關鍵的問題。
- 不要為了補齊所有欄位而提問。
- 不要把多個缺口硬合併成一個複合問題。

如果有多個缺口，寧可問兩個簡單問題，也不要問一個難回答的長問題。

## 好問題範例

- 請補充目前完成程度，例如「已完成登入頁調整，剩 Windows Codex 路徑測試」。
- 請補充目前是否如期，例如「目前仍符合時程」或「已延後，主要影響上線測試」。
- 請補充下週預計交付物，例如「下週預計完成 ClickUp 上傳測試」。
- 請補充是否需要主管協助，例如「目前無需主管介入」或「需要主管協調權限」。
- 請補充風險影響，例如「若 Codex CLI 設定未解，會影響 Windows 測試完成時間」。

## 不好的問題範例

- 目前專案健康度如何？
- 是否仍能依原訂時程推進？若不能，是否影響里程碑或需要主管協助？
- 是否有 Delay、阻塞、風險或需要主管介入？
- 下一步是什麼、何時完成、完成標準為何？

## 輸出要求

產生問題時：

- `question` 必須是使用者可以直接回答的具體問題。
- `question` 必須包含「請補充...」或等價語氣。
- `question` 必須包含一個簡短回答範例。
- `reason` 請輸出「參考範例」文字，讓使用者可以直接照格式回答。
- `reason` 不要寫管理判斷說明；應寫成例如「目前仍符合時程」或「已延後，主要影響上線測試」。
- `priority` 只能是 `high`、`medium` 或 `low`。
- `target_field` 必須是 `this_week_progress`、`next_week_plan`、`project_name` 或 `notes`。
- 不要產生抽象、複合、需要使用者自行分析管理意涵的問題。

## 停止追問條件

若目前內容已足以讓主管知道：

- 目前做到哪裡。
- 下一步是什麼。
- 是否如期或是否有明確風險。
- 是否需要主管協助。

即使仍可補充更多背景，也不要再產生追問。
