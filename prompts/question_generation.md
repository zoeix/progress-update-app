# 追問策略設計（Manager Readiness）

## 核心原則

追問的目的不是補齊所有欄位，也不是讓資訊越完整越好。

而是補足足以讓主管完成管理決策的重要資訊。

每一個問題都應至少幫助主管回答其中一項：

* 專案目前做到哪裡？
* 下一步是什麼？
* 是否需要主管介入？
* 是否影響時程？
* 是否影響里程碑？

若問題的答案不會提升主管對專案狀態的判斷能力，則不要提問。

---

# 問題優先順序

追問時，應依照下列順序決定詢問內容：

1. 是否影響交付或里程碑
2. 完成程度或目前狀態
3. 下一步交付物
4. Risk 與阻塞
5. 量化資訊
6. Tag 或格式問題

不要優先詢問對管理決策沒有幫助的資訊。

---

# Priority 定義

## High

回答後可以直接提升主管判斷能力。

例如：

* 完成程度
* 是否 Delay
* 是否影響里程碑
* 是否需要主管介入
* Risk 成因與影響

---

## Medium

回答後可以補足專案背景，但不會直接改變主管決策。

例如：

* 交付物名稱
* 影響模組
* 負責人
* 相依單位

---

## Low

回答後僅增加背景細節，不影響管理判斷。

例如：

* 技術實作方式
* 使用哪個 Framework
* Debug 過程
* 開發細節

---

# 追問數量原則

* 最多追問 3 題
* 能只問 1 題就不要問 2 題
* 能合併就不要拆開
* 每一題都應有明確目的

例如：

不要：

* API 完成多少？
* 是否 Delay？
* QA 做了嗎？

可以改成：

> API 目前完成到哪個階段？是否仍能依原定時程交付？

---

# 一題一目的

每個問題應只有一個主要目的。

避免同時詢問多個不同主題，增加回答負擔。

問題應讓使用者可以快速回答，而不是需要重新整理整份週報。

---

# 必問原則

應依據資訊是否足以支援管理決策，而不是單純檢查是否出現某些關鍵字。

例如：

若主管仍無法判斷：

* 做到哪裡
* 下一步
* 是否 Delay
* 是否需要主管介入

即使沒有使用：

* 進行中
* 討論中
* 處理中

仍應追問。

反之：

> API 開發進行中，目前已完成 CRUD，等待 QA 驗收。

雖然包含「進行中」，但主管已能判斷目前狀態，因此不需要追問。

---

# 不需要追問的情況

不要追問只會增加背景知識，但不影響管理決策的資訊。

例如：

* 使用哪個 Framework
* 如何實作
* 中間嘗試過哪些方法
* Debug 過程

若答案不會影響：

* 專案狀態
* 下一步
* Risk
* 時程
* 管理決策

則不要提問。

---

# 合併追問原則

若一個問題可以同時補足多個缺漏，應優先合併。

例如：

不要：

* 完成多少？
* 何時完成？
* 是否影響時程？

可以改成：

> 目前完成到哪個階段？是否仍能依原定時程交付？

---

# 停止追問條件

若目前資訊已足以讓主管：

* 知道目前專案狀態
* 知道下一步工作
* 判斷是否需要介入
* 判斷是否影響時程或里程碑

即使仍有可補充資訊，也不要再產生新的問題。

追問應在資訊足以支援管理決策時停止，而不是追求所有欄位都完整。

---

# 最終原則

追問的目的不是蒐集所有資訊，而是補足主管做管理決策所需要的最少資訊。

若目前內容已足以讓主管了解專案狀態、判斷是否需要介入，則不要因資訊仍可更完整而繼續追問。

高品質的追問，應讓使用者用最少的回答成本，補足最有價值的資訊。
